# Prompt для Orchestrator: реализация TRI-114 route covers

Ты работаешь как технический оркестратор реализации задачи TRI-114 в монорепозитории `travel-planner`.

## Контекст

Исходный утвержденный план находится в файле:

- `travel-planner/plans/trip-image-pixabay-plan.md`

Опирайся на этот план как на единственный источник требований к архитектуре и порядку внедрения.

## Цель реализации

Реализовать backend-механику автоназначения обложки маршрута в `trips.img`:

1. выбрать город по координатам подходящей точки маршрута,
2. транслитерировать город в латиницу,
3. найти локальную картинку в `apps/web/public/assets/images`,
4. если локально нет — получить картинку из Pixabay, сохранить локально,
5. записать web-путь в `trips.img`.

Важно: пересчет запускать только после изменений в `create/update/remove` точек. Для `reorder` пересчет НЕ делать.

## Ограничения

- Не ломать существующие API-контракты и UI.
- Изменения должны быть минимальными и локальными.
- Ошибки внешних сервисов не должны ломать основной CRUD точек.
- Секреты не хардкодить.

## Что нужно сделать обязательно

### 1) Сформировать детальные Acceptance Criteria

Перед кодингом создай полный список Acceptance Criteria с проверяемыми условиями, минимум в блоках:

- Functional
- Reliability and fallback
- Security
- Performance and resilience
- Non-regression

Каждый критерий должен быть тестируемым и иметь однозначный ожидаемый результат.

### 2) Реализовать задачу по плану

Соблюдай архитектуру из `travel-planner/plans/trip-image-pixabay-plan.md`.

Обязательно реализуй:

- выделенный сервис доменной логики для image resolution,
- определение города по reverse geocoding от координат,
- нормализацию/transliteration в slug,
- локальный lookup файлов изображений,
- Pixabay fallback с корректной обработкой ошибок,
- безопасное сохранение изображения,
- идемпотентное обновление `trips.img`,
- вызовы пересчета только из `PointsService.create/update/remove`.

### 3) Разработать smoke UX tests через curl

После реализации составь и выполни смоки пользовательского опыта через `curl`.

Смоки должны покрывать минимум:

1. Создание маршрута и добавление точки -> `trips.img` установлен.
2. Локальный файл найден -> путь в `trips.img` указывает на `/assets/images/...`.
3. Локальный файл отсутствует -> Pixabay fallback отрабатывает и картинка сохраняется локально.
4. Ошибка Pixabay или пустой ответ -> API не падает, старый `img` не затирается.
5. `reorder` точек -> `trips.img` не пересчитывается и не меняется только из-за reorder.
6. `update/remove` точки -> пересчет запускается и поведение соответствует критериям.

Для каждого smoke-сценария предоставь:

- цель теста,
- команды `curl`,
- фактический ответ,
- короткий verdict PASS/FAIL.

## Формат итогового отчета

В конце выдай структурированный отчет:

1. Что реализовано
2. Какие файлы изменены
3. Acceptance Criteria (чеклист)
4. Прогон smoke UX tests через curl (с выводами)
5. Риски/ограничения, если остались

Без отчета и без прохождения smoke-тестов задача считается незавершенной.
